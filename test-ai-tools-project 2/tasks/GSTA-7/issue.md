Assumptions
-----------
- Multi-tenant: tenant isolation is required; tenants belong to an Organization.
- Core entities: Organization, User, Project, Resource (generic domain object), Event/Audit logs.
- Postgres is the canonical datastore. We will use UUIDs, JSONB for extensible fields, and soft deletes where appropriate.

High-level design decisions
--------------------------
- Keys and identity: use UUIDv4 primary keys (column type: uuid) generated by application or database (gen_random_uuid()).
- Timestamps: all tables get created_at (timestamptz) and updated_at (timestamptz) with triggers or app-managed updates.
- Tenancy: Organization-level tenancy enforced by a non-null organization_id FK on tenant-owned tables. Row-level security (RLS) will be considered in Phase 3 if stricter isolation required.
- Extensibility: use JSONB "metadata" for attributes that are product-volatile or optional to avoid frequent migrations.
- Auditing: immutable event/audit log table to store important state transitions and who performed them. Use append-only patterns.
- Indexing: create composite indexes for common query patterns (organization + created_at, user email unique per org, project name per org, etc.).

Data flow / sequence (common flows)
----------------------------------
1. User sign-up
   - Create Organization (if new), create User (owner role), create default Project(s)
   - Emit AuditLog entries for org/user creation

2. Resource lifecycle
   - Create Resource (belongs to Project), update Resource, soft-delete Resource
   - Emit AuditLog for create/update/delete with before/after snapshots in JSONB

3. Query patterns
   - List resources by project with pagination ordered by created_at
   - Search resources by indexed attributes
   - Retrieve audit history for a resource

Migration strategy
------------------
1. Create migration scripts that are forward-compatible (additive) and backwards-compatible when possible.
2. For destructive changes (drop column/table, change column type), use a 2-step migration:
   - Step A: Add new column + backfill in a migration; deploy app code that writes/reads both old and new columns.
   - Step B: Switch reads to new column only, then in a later release remove the old column.
3. Backups: take a consistent logical or physical backup before each major migration. Test restore to a staging cluster.
4. Rollback: if a migration fails, rollback using backup/restore. For additive-only migrations, rollback is typically a schema downgrade by removing columns (careful with data loss).

Failure modes and mitigation
----------------------------
- Migration failures: mitigate via pre-run on staging with production-sized data snapshot and rehearsal of rollback.
- Long-running migrations: use batching and background jobs to backfill; avoid table-locking operations on large tables.
- FK constraint errors: validate referential integrity during staging; pre-populate missing parents if the production data is inconsistent.
- Hotspots/Hot partitions: avoid sequential integer PKs, use UUIDs; add sharding/partitioning plan later if table sizes exceed single-node capacity.

Security and trust boundaries
----------------------------
- Do not store secrets in plain text; use encrypted secrets in application layer or dedicated secrets manager.
- Principle of least privilege for DB users: read-only replicas, limited write-role for app, separate migration role for schema changes.
